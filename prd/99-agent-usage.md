# 99) “PRD 사용량” 관점 정리(에이전트/작업자용)

여기서 말하는 “사용량”은 웹 유저 트래킹이 아니라,
**작업 중 에이전트들이 PRD를 계속 읽고(참조) 수정하면서 PRD 의존성과 변경량이 커지는 현상**을 관리하는 방식이다.

## 목표
- 어떤 작업이 PRD의 어떤 섹션에 의존했는지 추적(= “무엇을 읽고 작업했는지”)
- PRD 변경이 생길 때 영향 범위를 빠르게 파악(= “어디가 바뀌었는지”)
- 결정/가정을 문서에 고정해 반복 질문/재작업 최소화

## 원칙(작업 흐름)
1) 작업 시작: 필요한 PRD 섹션 파일만 읽는다(전체를 매번 정독하지 않기)
2) 작업 중: 수정/결정/가정이 생기면 즉시 기록한다
3) 작업 종료: `WORKLOG.md`에 “읽은/수정한 파일”을 남긴다(짧아도 됨)

## 언제 무엇을 기록하나

### A) WORKLOG (사용량: 읽고/고친 흔적)
- 위치: `WORKLOG.md`
- 남기는 내용:
  - 읽은 PRD 파일(의존성)
  - 수정한 PRD 파일(변경)
  - 생성/변경 산출물(코드/디자인/카피/이미지 등)
  - 오픈 질문/리스크

### B) DECISIONS (결정: 되돌리기 쉽게)
- 위치: `DECISIONS.md`
- 남기는 내용:
  - “결정” + “왜” + “대안”
  - 예: “v1 CTA는 전화+카카오+폼” / “디자인 톤은 블랙+골드”

### C) PRD 섹션 파일(요구사항의 소스 오브 트루스)
- 실제 요구사항 변경은 해당 섹션 파일에 반영한다.
- 값이 없으면 `{{PLACEHOLDER}}`를 유지한다(추측으로 채우지 않기).

## 섹션 단위로 작업을 쪼개는 방법(추천)

### 예시 1) “CTA 버튼 스타일 구현” 작업
- 읽기: `05-nfr-design.md`, `01-ia-sections.md`
- 수정: (필요 시) `05-nfr-design.md` 토큰/버튼 규칙 보완
- 기록: `WORKLOG.md`에 “CTA 버튼 토큰/상태 구현, 어떤 섹션을 근거로 했는지”

### 예시 2) “문의 폼 백엔드(시트 적재)” 작업
- 읽기: `02-form.md`, `06-tech-release-ops.md`
- 수정: (필요 시) 시트 컬럼/운영 정책 확정
- 기록: `WORKLOG.md` + 결정이 생기면 `DECISIONS.md`

## 최소 요구(강제 규칙)
- PRD 파일을 수정했으면 **반드시** `WORKLOG.md`에 해당 엔트리를 남긴다.
